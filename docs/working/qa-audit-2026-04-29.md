# QA-Audit Heizungssteuerung Hotel Sonnblick

**Datum:** 2026-04-29
**Branch im Audit:** `feat/sprint6-hardware-pairing` (lokal) + Live-Stand `:develop`-Tag auf `heizung-test.hoteltec.at`
**Letzter Tag deployed (Test):** `v0.1.5-lorawan-foundation`
**Auditor:** QA-Pass (statisch + destruktiv gegen Test-Server)
**Umfang:** Vollaudit (Code, API, DB, Infra, Tests, GitHub-Hygiene)
**Schweregrad-Schema:** K = kritisch, H = hoch, M = mittel, N = niedrig, I = info

---

## 0. Executive Summary

Das Projekt ist solide aufgesetzt: klare ADR-Disziplin, ehrliche Sprint-Doku, getrennte Test-/Main-Stages, GHCR-Pull-Deploy, Branch-Protection. Codequalität (Typing, Pydantic, SQLAlchemy 2.0-Style, Async) ist überdurchschnittlich. Aber:

- **Es gibt keine Authentifizierung.** Alle API-Endpoints — inklusive `/openapi.json`, `/docs`, `/redoc` und (sobald Sprint 6 deployt ist) `POST /api/v1/devices` — sind öffentlich. Jeder im Internet kann sehen, was ihr baut, und sobald CRUD live ist auch schreiben. Das ist der mit Abstand wichtigste Befund (K-1).
- **Die Heizungs-Regel-Engine existiert noch nicht.** Frostschutz, Vorheizen, Nachtabsenkung — alles im Strategie-Dokument, aber `backend/src/heizung/rules/` enthält nur ein leeres `__init__.py`. Das ist legitim für den aktuellen Sprint-Stand, muss aber als Realität anerkannt werden: das System steuert heute nichts (M-13).
- **Ein Crash-Pfad ist live:** `GET /api/v1/devices/9999999999999999999/sensor-readings` liefert 500 Internal Server Error (PostgreSQL int4-Overflow). Reproduzierbar, ungetestet (K-2).
- **STATUS.md ist optimistischer als die Wirklichkeit.** Sprint 6.10 (Devices-CRUD) ist als „erledigt" markiert, aber auf `develop` nicht gemerged → Test-Server kennt nur den Sprint-5-Stand. Frontend `/devices` läuft auf 404 (H-1).
- **Letzter Deploy-Test-CI-Run auf `develop` ist `failure` (28.04, SHA `e4b4632`)** und wurde nicht behandelt (H-2).
- **Test-Coverage ist asymmetrisch:** 32 Backend-Tests prüfen Pydantic-Schemata und Importbarkeit, aber **kein einziger Test schlägt eine echte HTTP-Route gegen eine echte DB** (H-4).

Bewertung: **MVP-tauglich für interne Vorzeige-Demo, NICHT bereit für Hotel-Produktivbetrieb.** Die Gap-Liste ist überschaubar, aber jede einzelne Lücke unter „Kritisch" und „Hoch" ist Show-Stopper für reale Heizungssteuerung.

## 2. Befunde nach Schweregrad

### 2.1 KRITISCH (Show-Stopper, vor Produktion zwingend)

#### K-1 · Keine Authentifizierung an der API
**Reproduktion:**
```bash
curl https://heizung-test.hoteltec.at/openapi.json   # 200, vollständige API-Doku
curl https://heizung-test.hoteltec.at/docs           # 200, Swagger-UI
curl https://heizung-test.hoteltec.at/api/v1/devices/1/sensor-readings   # 200/404, ohne Header
```
**Impact:** Sobald Sprint 6 (Devices-CRUD) auf `develop` gemerged ist, kann jeder im Internet mit `POST /api/v1/devices` Geräte anlegen, ändern, deaktivieren. Das System ist offen für Vandalismus und Datenmissbrauch. Bei späteren Setpoint-Endpoints (Heizungs-Steuerung) wäre das ein direkter Eingriff in den Hotelbetrieb durch Fremde.
**Fix:**
1. Sofort: API-Key-Auth (HTTP-Header `X-API-Key`) als Minimal-Mittel — passt zu AE-12. ENV-Variable `API_KEYS` (komma-getrennte Liste).
2. Mittelfristig: User-Session-Auth (NextAuth oder FastAPI-Users) für UI, separate API-Keys mit Scopes für Maschinen-Zugriff.
3. `/openapi.json`, `/docs`, `/redoc` im Production-Mode hinter Auth oder via Caddy-Basic-Auth abschirmen.

#### K-2 · 500 Internal Server Error bei großem `device_id`
**Reproduktion:**
```bash
curl https://heizung-test.hoteltec.at/api/v1/devices/9999999999999999999/sensor-readings
# → HTTP 500: "Internal Server Error"
```
**Ursache:** Wert > Postgres `int4`-Range; `await session.get(Device, device_id)` wirft `DataError`, FastAPI hat keinen globalen Exception-Handler → Bare Stacktrace im Log, generische 500.
**Impact:** Information-Leakage-Risiko, schlechte UX für API-Clients, Blueprint für Bot-Scanner.
**Fix:** Path-Param mit `Path(..., gt=0, le=2_147_483_647)` validieren, plus globaler Exception-Handler für `sqlalchemy.exc.DBAPIError` → 422/404.

#### K-3 · Schwache Default-Secrets im `.env.example` und Compose
**Befund:**
- `SECRET_KEY=change-me-in-production` (default in `config.py`, Validator triggert nur bei `ENVIRONMENT=production`)
- `MQTT_HEIZUNG_PASSWORD=changeme`, `MQTT_CHIRPSTACK_PASSWORD=changeme` in `.env.example`
- `CHIRPSTACK_DB_PASSWORD=chirpstack` (Default in Compose)
- `CHIRPSTACK_API_SECRET=please-change-this-secret-min-32-chars-long` (Default)
**Impact:** Wenn auf einem Server `ENVIRONMENT` nicht gesetzt ist (oder `=development`), bleibt der Default-`SECRET_KEY` aktiv. ChirpStack-DB-Passwort und Mosquitto-Defaults sind im öffentlichen Repo dokumentiert.
**Fix:**
1. Validator in `config.py` erweitern: alle Secrets mit Default-Werten dürfen NIE in irgendeinem ENVIRONMENT laufen, nicht nur in `production`. Ausnahme: lokale Entwickler-Maschinen via Marker `ALLOW_DEFAULT_SECRETS=1`.
2. `.env.example` so umbauen, dass die Default-Werte explizit `<bitte-setzen>` sind und Compose-Start ohne echten Wert fehlschlägt.
3. ChirpStack-Compose: `CHIRPSTACK_DB_PASSWORD` ohne Default, beim Compose-Start zwingend.

#### K-4 · ChirpStack & Gateway-Bridge laufen als root
**Befund:** `infra/deploy/docker-compose.prod.yml` setzt `user: "0:0"` für `chirpstack-init`, `chirpstack-gateway-bridge-init`, `chirpstack-gateway-bridge` und `chirpstack`. Begründung im ADR (AE-Lessons-Learned): Permission-Issue mit Bind-Mount.
**Impact:** Container-Escape (zugegeben: derzeit nicht bekannt) hätte direkten Root-Zugriff auf den Host-Kernel-Namespace. Defense-in-Depth-Verletzung.
**Fix:** Statt `user: "0:0"` die Render-Volumes mit korrekten UIDs vorbereiten (init-Sidecar `chown 65532:65532 /output/*` o. ä.) und ChirpStack-Container als nonroot. Falls ChirpStack-Image keinen Nonroot-User hat, eigenes Wrapper-Image bauen.

#### K-5 · Keine Content-Security-Policy
**Befund:** Antwort-Header von `https://heizung-test.hoteltec.at/`:
```
strict-transport-security: …
x-content-type-options: nosniff
x-frame-options: DENY
referrer-policy: strict-origin-when-cross-origin
```
Kein `Content-Security-Policy`, kein `X-XSS-Protection`, kein `Cross-Origin-*`.
**Impact:** Wenn ein Sprint mal `dangerouslySetInnerHTML` einführt, ein Drittanbieter-Skript einbaut oder ein Codec-Decoder JS-Code rendern muss, ist XSS frei. Bei reinem React-Rendering moderat, aber kein Net-Layer-Schutz.
**Fix:** Caddy-Header in `Caddyfile.test` und `Caddyfile.main` ergänzen:
```
Content-Security-Policy "default-src 'self'; img-src 'self' data:; style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src 'self' fonts.gstatic.com; connect-src 'self'; frame-ancestors 'none'"
Cross-Origin-Opener-Policy "same-origin"
Cross-Origin-Resource-Policy "same-origin"
```
Abgleich mit Material-Symbols-Loading-Strategie (Google Fonts) und Recharts-Bedarf nötig.

#### K-6 · Frostschutz ist nicht implementiert
**Befund:** `backend/src/heizung/rules/__init__.py` enthält 0 LOC produktiven Code. Frostschutz steht in der Strategie als Systemkonstante (10 °C, nie überschreibbar), aber kein einziger Code-Pfad setzt diesen Wert irgendwo. Auch keine Edge-Fallback-Konfig (AE-05) ist umgesetzt.
**Impact:** Sobald echte Hardware paired ist (Sprint 6.7+), liegt die Verantwortung für Frostschutz beim Default-Setpoint des Vicki-Geräts oder gar nicht. Bei Cloud-Ausfall vor Implementierung der Edge-Fallback-Logik gibt es keinerlei Garantie.
**Status:** Bekannt geplant, aber **muss vor erstem Pairing-Live-Test als nicht-funktional kommuniziert werden**, sonst entsteht falsche Sicherheit.
**Fix:** Vor Sprint-6.7-Pairing mindestens die Hardcoded-Konstante einbauen + Vicki-Default auf 10 °C setzen + dokumentieren, dass Cloud-Regel-Engine noch leer ist.

### 2.2 HOCH (Sprint-Plan-relevant, vor nächstem Tag fixen)

#### H-1 · STATUS.md ↔ Deploy-Stand divergent
Auf `heizung-test.hoteltec.at` ist nur Tag `v0.1.5-lorawan-foundation` deployed. `feat/sprint6-hardware-pairing` ist nicht in `develop` gemerged → keine `/api/v1/devices`-CRUD-Routes, kein Frontend-Dashboard sichtbar.
`https://heizung-test.hoteltec.at/devices` liefert **Next.js-404** (englischsprachig, ohne Design-System).
**Fix:** Sprint-6 ASAP gegen develop mergen oder STATUS.md korrigieren ("im Branch fertig, nicht deployed"). Sonst entsteht falsche Erwartung beim Pairing-Tag.

#### H-2 · Failure-CI-Run wurde nicht behandelt
GitHub-Actions-Run `Deploy Test` auf `develop`-Push (SHA `e4b4632`, 2026-04-28 14:00) ist `conclusion: failure`. Dauer ~29 s → früher Abbruch (vermutlich SSH/Push-Workflow-Rest oder fehlendes Secret nach Sprint-6-Cleanup-Commit `199dd87 chore(sprint6): obsolete SSH-push-deploy-Workflows entfernt`).
**Fix:** Run-Logs prüfen. Wenn der Workflow tot ist (SSH-Push-Deploy ist laut RUNBOOK ersetzt), Workflow-Datei löschen statt fehlschlagen lassen. Required-Status-Checks gegen Toten-Workflow filtern.

#### H-3 · Frontend `/api/health` von extern unerreichbar
Caddy routet `path /api/* /openapi.json /docs /redoc /health` zur FastAPI. Der dedizierte Web-Container-Healthendpoint `frontend/src/app/api/health/route.ts` ist **nur Container-intern** erreichbar (Docker-Healthcheck OK), aber externes Monitoring kann ihn nicht sehen — der Pfad gehört dem Backend.
**Fix:** Im Backlog (STATUS.md §6) erwähnt. Vorschlag: Frontend-Healthendpoint auf `/_health` umbenennen (oder zusätzlich routen) + Caddy-Matcher anpassen.

#### H-4 · Keine API-Integration-Tests
32 Backend-Tests laufen, aber keiner trifft die echten HTTP-Endpoints mit echter DB-Persistenz:
- `test_health.py` testet nur `/health` und `/openapi.json` (in-Process FastAPI-TestClient)
- `test_device_schema.py` testet nur Pydantic-Validierung
- `test_models.py` testet nur Importierbarkeit & SQLAlchemy-Metadaten (ohne DB)
- `test_mqtt_subscriber.py` testet Pure-Functions, aber explizit nicht die DB-Persistenz
**Folge:** Mein 500er-Bug (K-2) wäre durch einen einzigen Test mit `client.get("/api/v1/devices/9999999999999999999/sensor-readings")` aufgefallen. Der CRUD-Pfad (POST/GET/PATCH) ist komplett ungetestet, obwohl als „erledigt" markiert.
**Fix:** Sprint einplanen. Empfohlen: `pytest-postgresql` oder `testcontainers-python` mit eigener Test-DB. Mindest-Coverage: pro API-Route Happy-Path + 1 Negative-Path.

#### H-5 · `.env`-Datei lokal mit `change-me-in-production`-SECRET_KEY
`.env` enthält den Default. Validator triggert nur bei `ENVIRONMENT=production` — bei lokalem Dev ist das OK, aber wenn diese `.env` versehentlich auf einen Server kopiert wird (was per CLAUDE.md ausgeschlossen sein sollte), blockt der Validator nur, falls `ENVIRONMENT=production` explizit gesetzt ist.
**Fix:** Im Lokal-Default `ENVIRONMENT=development` zwingend, im Server-Setup `ENVIRONMENT=production` als Pflichtfeld in `.env`. Pre-Boot-Check im Entrypoint: bricht ab, wenn `ENVIRONMENT` leer ist.

#### H-6 · `:develop`-Tag-Mutation ohne Versionsstabilität
GHCR-Tag `develop` wird bei jedem Push überschrieben. Pull-Deploy zieht „den neuesten develop". Kein Audit-Trail, was wann lief, kein Rollback auf konkrete Version möglich, keine Reproduzierbarkeit eines konkreten Stands. Bei Vorfall („gestern Mittag stieg die Heizung aus") nicht rekonstruierbar.
**Fix:** Zusätzlich zum mutierenden `develop`-Tag immer `develop-<sha>` taggen (build-images.yml macht das schon: `type=sha,prefix={{branch}}-,format=short`). Im Compose statt `develop` das SHA-tag pinnen, in `.env` als `IMAGE_TAG`. Deploy-Pull-Skript dokumentiert dann „lief auf `develop-e4b4632`".

#### H-7 · `frontend-ci-skip.yml`-Hack hebelt Required-Checks aus
Datei meldet "lint-and-build" und "e2e" als success ohne irgendetwas zu prüfen, sobald keine Frontend-Datei betroffen ist. Sicherheits-Risiko: Wer den Workflow unautorisiert anfasst, kann jeden PR „grün" stellen ohne CI.
**Fix:** Branch-Protection-Konfiguration auf "any of these checks must succeed" statt "all" umstellen, falls GitHub das mittlerweile unterstützt. Alternativ: Required-Checks reduzieren auf Workflows, die wirklich immer laufen (Backend-CI ohne Path-Filter machen).

#### H-8 · Kein dokumentiertes Backup-Verfahren
RUNBOOK kennt Rescue-Mode, UFW, GHCR-PAT, DNS — aber kein Wort zu `db_data`-Volume-Backup. Bei Hetzner-Hardware-Ausfall oder versehentlichem `docker volume rm` sind alle Sensor-Readings, Devices, Occupancies weg.
**Fix:** RUNBOOK §11: tägliches `pg_dump` per systemd-Timer auf `/opt/backups/`, retention 30 Tage, monatlicher Off-Site-Sync (Hetzner Storage Box). TimescaleDB hat eigene Backup-Hinweise (Continuous Aggregates, Compressions-aware) — beachten.

## 5. Empfehlungen, priorisiert

### 5.1 Sofort (vor Pairing-Tag oder bis morgen)
1. STATUS.md auf den **echten** Deploy-Stand korrigieren oder Sprint-6-Branch in `develop` mergen.
2. Failure-CI-Run analysieren und Workflow-Datei löschen oder fixen (`Deploy Test`).
3. Vor Pairing dem IT-Mitarbeiter klar kommunizieren: Frostschutz-Logik + Regel-Engine sind **leer**, das System steuert nichts. Pairing-Tag liefert Sensor-Werte, keine Aktorik.

### 5.2 Diese Woche (vor erstem Heizungs-Steuer-Sprint)
1. **K-1: Auth einbauen.** Minimal-Variante: API-Key-Header in einer FastAPI-Dependency. Sprint-Brief 1 Tag, Implementierung 2 Tage.
2. **K-2: Globaler Exception-Handler** + `Path(..., gt=0)` auf allen ID-Pfaden.
3. **K-3: Secrets-Validator härten** (alle Defaults verbieten ohne `ALLOW_DEFAULT_SECRETS=1`).
4. **K-5: CSP-Header** in Caddyfiles (gemeinsames Snippet).
5. **H-4: Erste API-Integration-Tests** (mindestens Devices-CRUD).

### 5.3 Nächster Sprint
1. **K-6: Frostschutz hardcoded** als Konstante einziehen + Vicki-Default-Setpoint setzen + Edge-Fallback nach AE-05.
2. **H-6: SHA-pinning für GHCR-Tags** + IMAGE_TAG in `.env` versionieren.
3. **H-8: Backup-Strategie** (`pg_dump` + Retention + Off-Site).
4. **K-4: ChirpStack ohne root** (eigenes Wrapper-Image oder Volume-Permissions korrigieren).

### 5.4 Ständige Pflege
- Test-Coverage in CI als Required-Check.
