# Sprint 8.15 — Design-Strategie-Konformität (Hotfix)

**Datum:** 2026-05-03
**Branch:** `chore-sprint8-15-design-fixes`
**Tag (geplant nach Merge):** `v0.1.8-stammdaten`

## Anlass

User hat 3 UI-Bugs nach Sprint 8 gemeldet:

1. Umlaute werden falsch angezeigt — alle deutschen Strings im Frontend nutzen ASCII-Workarounds („Loeschen", „Aenderungen", „Fuer", „Naechste", „Geraet").
2. Farb-Konsistenz fehlt — Submit-Buttons in Rosé statt Grün, Lösch-Aktionen als Text-Link statt rotem Outline-Button.
3. Schriftgröße zu klein — Body-Text 13–14 px, Strategie schreibt 14–16 px vor.

## Recherche-Ergebnis

Alle 3 Punkte sind belegt durch Design-Strategie 2.0.1:

- **§3.2 Schriftgrößen-Skala:** `--text-base` = 16 px, `--text-sm` = 14 px, `--text-xs` = 12 px. Code hatte 14/13/11 — eigenmächtige Abweichung „kompakte Hotelier-Sicht" ohne Strategie-Deckung.
- **§6.1 Add-Button (grün):** `--color-add` (#16A34A) ausschließlich für „Neu anlegen"-Aktionen, immer mit Plus-Icon links.
- **§6.1 Destructive Button:** Outline mit `--color-error` als Text und Rahmen, Hover `--color-error-soft`, **immer mit Bestätigungs-Dialog mit zweitem Klick-Schutz**.
- **§6.1 Primary Button:** Rosé bleibt für „Speichern" einer existierenden Sache (Update-Aktionen) — Strategie sagt Primary ist Haupt-CTA pro Region, max. 1.

Bug 1 ist ebenfalls strategiefremd — UTF-8 funktioniert in Next.js sauber, ASCII-Workarounds waren technische Vorsicht ohne realen Nutzen.

## Umsetzung

### Token-Layer (`globals.css` + `tailwind.config.ts`)

- Schriftgrößen auf 12/14/16/18/20/24/30/36 px gesetzt (Strategie §3.2). Body-Standard hebt sich von 14 → 16 px.
- Neue Tokens `--color-add`, `--color-add-hover`, `--color-on-add` (#16A34A / #15803D / #FFFFFF).
- Semantik-Farben auf Strategie-Werte angeglichen: `--color-success` #16A34A, `--color-warning` #F59E0B, `--color-error` #E05252, `--color-info` #3B82F6 (vorher abweichend).
- Aliase `--color-danger` → `--color-error` für Sprint-7-Code-Kompatibilität.
- Tailwind-Mapping erweitert um `bg-add`, `text-on-add`, `text-error`, `bg-error-soft` und Top-Level `text-on-primary`/`text-on-add`.

### Neue UI-Komponenten

- **`components/ui/button.tsx`** — Button mit Varianten `primary`/`add`/`secondary`/`destructive`/`ghost`, Größen `sm`/`md`/`lg`, optionalem Material-Symbol-Icon links, Loading-State.
- **`components/ui/confirm-dialog.tsx`** — Modaler Bestätigungs-Dialog mit Fokus auf Abbrechen-Button, ESC-zum-Schließen, Backdrop-Klick. Pflicht für alle Destructive-Aktionen (Strategie §6.1).

## Risiken

- Schriftgrößen-Erhöhung um ~14 % kann Layouts brechen (Spalten zu eng, Tabs umbrechen). Browser-Test deckt das auf.
- Add-Farbe ist identisch mit `--color-success` und `--color-heating-on` — gewollt (Strategie nutzt #16A34A für „aktiv/erfolgreich/grün"), aber falls jemand „Add ≠ Success" trennen will, muss das ein eigenes Token werden.
- Confirm-Dialogs greifen jetzt auf 4 Stellen (Raumtyp/Zimmer/Heizzone/Belegung). Kein E2E-Test deckt sie ab — soll in einem späteren Sprint nachgezogen werden.
