## B — Backend: Engine — wo wird Sommermodus heute gelesen?

### B1 Engine-Layer 0

Funktion: `layer_summer_mode(ctx: _RoomContext) -> LayerStep` in
`backend/src/heizung/rules/engine.py:142-173`.

Die Stelle, die `summer_mode_active` liest:

```python
def layer_summer_mode(ctx: _RoomContext) -> LayerStep:
    ...
    if ctx.summer_mode_active:
        return LayerStep(
            layer=EventLogLayer.SUMMER_MODE_FAST_PATH,
            setpoint_c=int(FROST_PROTECTION_C),
            reason=CommandReason.SUMMER_MODE,
            detail="summer_mode_active=true",
        )
    return LayerStep(
        layer=EventLogLayer.SUMMER_MODE_FAST_PATH,
        setpoint_c=None,
        reason=CommandReason.SUMMER_MODE,
        detail="summer_mode_inactive",
    )
```


### B2 Weitere Backend-Stellen

Grep `summer_mode_active` im Backend liefert nur Engine, Schema, Model,
Migration, Tests (siehe A3-Tabelle). **Keine andere Service-Layer und
kein Subscriber/Job schreibt `summer_mode_active`.** Schreibzugriff
erfolgt heute ausschließlich über `PATCH /api/v1/global-config`.

### B3 Layer-0-Verhalten bei `summer_mode_active=true`

- **Setpoint:** `int(FROST_PROTECTION_C)` = `10` (aus
  `rules/constants.py`).
- **Reason:** `CommandReason.SUMMER_MODE`.
- **Trace:** `EventLogLayer.SUMMER_MODE_FAST_PATH`,
  `detail="summer_mode_active=true"`.
- **Pipeline-Effekt:** `evaluate_room` (`engine.py:795-804`) ruft
  `layer_summer_mode` immer auf; bei `ctx.summer_mode_active` wird
  Layer 1–4 übersprungen und sofort `layer_clamp` angewendet. Result
  hat `layers=(summer, clamp)`. Das Engine-Decision-Panel sieht damit
  2 Trace-Einträge statt 6.
