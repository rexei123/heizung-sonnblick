# Sprint 6.6.2 — Deploy-Pull-Skript: git-Sync ergaenzen

**Typ:** Infrastruktur / Operations
**Ziel:** `infra/deploy/deploy-pull.sh` synct vor jedem Deploy den Working-Tree gegen den passenden Branch und ruft danach `docker compose up -d` fuer ALLE Services auf — nicht nur api+web.
**Branch:** `chore/deploy-pull-git-sync`
**Geschaetzte Dauer:** 30–45 Min inkl. PR + Test-Deploy.

---

## Feature-Brief (Phase 1)

### Ausgangslage

Beim Pairing am 2026-04-29 hat der `heizung-test`-Server eine Compose-File-Aenderung (Mosquitto Port 1883 public) **nicht** ausgerollt. Root Cause: das deploy-pull-Skript zog nur die App-Images (`api`, `web`) aus GHCR, machte aber **kein** `git pull` und **kein** `docker compose up -d` fuer die Infra-Container.

Folge: Server-Working-Tree stand seit dem Vortag auf `feat/sprint6-hardware-pairing` (per Hand-Checkout fuer den Pairing-Tag), `origin/main` brachte alle PRs vorbei, aber das Skript ignorierte sie still. Mosquitto lief weiter mit der alten Compose-File-Version (kein Port-Mapping).

Das ist ein **wiederkehrendes Risiko**: jede Sprint-Pause auf einem Server-Branch droht, Compose-/Caddyfile-/Mosquitto-Aenderungen zu verschlucken. Auf `heizung-main` wartet derselbe Bug.

### Ziel — End-State Sprint 6.6.2

`deploy-pull.sh` macht drei Phasen:

1. **Working-Tree-Sync**: `git fetch + checkout + reset --hard` auf den passenden Branch (Mapping per `STAGE` in `.env`: `test→develop`, `main→main`, override via `DEPLOY_BRANCH`).
2. **Image-Pull**: `docker compose pull api web`.
3. **Container-Up**: `docker compose up -d --remove-orphans` ohne `--no-deps`. Recreate erfolgt nur bei Config- oder Image-Drift, sonst bleibt alles laufend.

Sicherheitsnetz: lokale Aenderungen am tracked Content brechen das Skript ab — kein silentes `reset --hard` ueber Hand-Hotfixes.

### Out of Scope

- Rollback-Mechanismus (separater Sprint, siehe Sprint-8-Backlog).
- Health-Check-Probing nach Deploy (Caddy `/api/v1/health` ist da, aber Skript prueft es nicht aktiv).
- Slack-/Mail-Notification bei Deploy-Fehler.

### Technische Annahmen

- `infra/deploy/.env` existiert auf jedem Server und enthaelt `STAGE=test` oder `STAGE=main`.
- `/opt/heizung-sonnblick` ist auf jedem Server ein git-Checkout mit Remote `origin` (https, public-read).
- Server hat `safe.directory` fuer `/opt/heizung-sonnblick` gesetzt (siehe RUNBOOK §6).

## Edge Cases

- **Lokale Aenderungen am Server:** Skript bricht ab und logged `git status --short`. User muss manuell stashen oder committen.
- **Rebase/Force-Push auf origin/develop:** `reset --hard origin/develop` zieht in jedem Fall die origin-Version (kein Merge-Konflikt moeglich).
- **`origin/<branch>` existiert nicht:** Skript bricht in Phase 1 mit `git rev-parse origin/<branch>` ab, kein Container-Restart.
- **Compose-File hat Syntax-Fehler nach Pull:** `docker compose up -d` schlaegt fehl, alte Container laufen weiter (kein partieller Restart).
- **DEPLOY_BRANCH zeigt auf nicht-existierenden Branch:** Skript bricht ab, kein Container-Restart.

---

## Risiken

| Risiko | Eintritt | Impact | Mitigation |
|---|---|---|---|
| `reset --hard` zerstoert Hand-Hotfixes | gering (Skript checkt vorher) | hoch | `git diff --quiet HEAD`-Check vor reset |
| Recreate aller Container fuehrt zu unerwartetem Restart | mittel (erstmals nach Deploy) | mittel | `up -d` ohne `--force-recreate`, nur Drift triggert Restart |
| ChirpStack-Init-Sidecar laeuft ungewollt nochmal | gering | gering | Init-Container haben `restart: "no"`, idempotent |
| Schluesselrotation der GHCR-PAT ueberraeger Deploy | gering | hoch | Skript bricht in Phase 2 ab, alte Container laufen |

## Verifikation

Nach Deploy auf `heizung-test`:

```bash
sudo systemctl start heizung-deploy-pull.service
sudo journalctl -u heizung-deploy-pull.service -n 50 --no-pager | grep -E 'Sync|HEAD|Fertig'
cd /opt/heizung-sonnblick && git log --oneline -1   # muss origin/develop-HEAD sein
cd /opt/heizung-sonnblick && git status              # nothing to commit, working tree clean
```

Erwartung: `Sync feat/...@<sha1> -> develop@<sha2>` beim ersten Lauf, danach `Working-Tree bereits auf origin/develop`.
