## Ausgangslage

Beim Pairing-Tag (2026-04-29) deployte der `heizung-test`-Server eine Compose-Aenderung (Mosquitto Port 1883 public) **nicht** aus. Root Cause: Das `deploy-pull.sh`-Skript zog nur die App-Images (`api`, `web`) aus GHCR, machte aber **kein** `git pull` und **kein** `docker compose up -d` fuer die Infra-Container (mosquitto, chirpstack, caddy).

Folge: Server-Working-Tree stand auf einem alten Feature-Branch, Compose-Schema-Aenderungen kamen nie auf den Server. Auf `heizung-main` wartete derselbe Bug.

## Aenderungen

### `infra/deploy/deploy-pull.sh` — Drei-Phasen-Logik

1. **Working-Tree-Sync** — `git fetch + checkout + reset --hard` auf den passenden Branch (Mapping per `STAGE` in `.env`: `test → develop`, `main → main`, override via `DEPLOY_BRANCH`).
2. **Image-Pull** — `docker compose pull api web` aus GHCR (mutierender Tag `develop`/`main`).
3. **Container-Up** — `docker compose up -d --remove-orphans` ohne `--no-deps`. Recreate erfolgt nur bei Config- oder Image-Drift.

Sicherheitsnetz: Lokale Aenderungen am tracked Content brechen das Skript ab — kein silentes `reset --hard` ueber Hand-Hotfixes.

### `infra/deploy/.env.example`

`DEPLOY_BRANCH` (kommentiert) als Override fuer das STAGE-Mapping ergaenzt.

## H-6 SHA-Pinning: Versuch + Revert

Drei Iterationen versucht (PRs #16, #18, #20), alle revertiert via PR #24:

1. **Version 1:** `IMAGE_TAG=<branch>-$(git rev-parse --short HEAD)` — schlug fehl weil HEAD-SHA bei Infra-only-PRs keinen Image-Build hat.
2. **Version 2:** `IMAGE_TAG=<branch>-$(git log -1 -- backend/ frontend/ ...)` — schlug fehl weil `build-images.yml` mit GitHub-Push-Event-SHA (= Merge-Commit) taggt, der `git log` aber den Source-Branch-Commit findet. Verschiedene SHAs bei `gh pr merge --merge` → Tag-Mismatch → Pull schlaegt fehl.

**Konsequenz:** SHA-Pinning erfordert sowohl `build-images.yml`-Anpassung (zusaetzlich Source-SHA als Tag) als auch `deploy-pull`-Logik. Eigener Sprint, nicht im Pairing-Wartemodus. Backlog Task #70.

## Lessons Learned (in CLAUDE.md verewigt)

- **EIN Thema → EIN PR.** Drei Hardening-Items (git-Sync, H-3, H-6) in einen PR zu packen war der Fehler. Stattdessen sequenziell, mit Server-Verifikation zwischen jedem Merge.
- **Cowork-Mount-Sandbox ist unzuverlaessig** fuer File-Sync zu Windows. Sandbox-Edits an neuen Files werden verschluckt. Bestehende Files bekommen Null-Bytes oder BOMs eingestreut. Konsequenz: Sandbox-Edit, sofort `git diff --stat` in PowerShell verifizieren, Cleanup falls Bin-Diff.
- **PS5 + Set-Content -Encoding UTF8 schreibt BOM** → Bash-Skripte brechen mit `/bin/bash: not found`. Sicherer Pfad: `[System.IO.File]::WriteAllText` mit `UTF8Encoding $false`. Skripte nach Moeglichkeit ASCII-only halten.
- **CI-Tagging-Verhalten verstehen vor SHA-Pinning-Logik.** GitHub-Push-Event-SHA != Source-Branch-Commit-SHA bei Merge-Strategie.
- **Im Pairing-Wartemodus keine Architektur-Aenderungen.** "Quick" Hardening-Items sind selten quick.
