# Sprint 6.6.4 — Mosquitto MQTT-Auth-Hardening (M-14)

**Typ:** Sicherheit / Operations
**Ziel:** Mosquitto laeuft auf den Production-Servern mit `allow_anonymous false`, `password_file` und `acl_file`. Drei MQTT-User mit Topic-getrenntem Berechtigungs-Scope.
**Audit-Bezug:** QA-Audit `docs/working/qa-audit-2026-04-29.md` Befund M-14.
**Datum:** 2026-04-30

---

## Ausgangslage

Sprint 6.6.1 (PR #13) hat den Mosquitto-Port 1883 auf den Test-Server-Public-IP geoeffnet, damit das UG65-Gateway im ChirpStack-v4-Modus direkt MQTT publishen kann. Mosquitto lief seitdem **anonymous** — jeder im Internet konnte publish/subscribe.

QA-Audit-Befund M-14: `mosquitto_sub` mit `-u heizung-api -P <pwd>` im Healthcheck legt das Passwort via `ps aux` offen + Mosquitto akzeptiert anonyme Verbindungen.

## Aenderungen

### `infra/mosquitto/config/mosquitto.prod.conf` (neu)

Production-Config mit `allow_anonymous false`, `password_file /mosquitto/config/passwd`, `acl_file /mosquitto/config/acl`. Lokale `mosquitto.conf` (anonymous, fuer Bind-Mount-Permission-Workaround unter Windows) bleibt unveraendert.

### `infra/mosquitto/config/acl`

Drei User mit Topic-getrenntem Scope:

| User | Read | Write |
|---|---|---|
| `chirpstack` | `application/#`, `gateway/#`, `eu868/#`, `$SYS/#` | `application/#`, `gateway/#`, `eu868/#` |
| `gateway-ug65` | `eu868/gateway/+/command/+` | `eu868/gateway/+/event/+`, `gateway/+/state/+` |
| `heizung-api` | `application/+/device/+/event/{up,join,status,ack}`, `$SYS/#` | `application/+/device/+/command/down` |

`gateway-ug65` ist neu (UG65 publisht direkt zum Mosquitto im ChirpStack-v4-Modus). `chirpstack` und `heizung-api` waren bereits in der ACL.

### `infra/mosquitto/config/passwd.example`

Aktualisierte Anleitung mit allen drei Usern + Verweis auf RUNBOOK §10a.

### `infra/deploy/docker-compose.prod.yml`

Mosquitto-Service: `command: mosquitto -c /mosquitto/config/mosquitto.prod.conf`. Damit liest Mosquitto die Production-Config statt der Default-`mosquitto.conf` (lokale anonymous). Healthcheck unveraendert (nutzt `MQTT_HEIZUNG_PASSWORD`).

### `infra/deploy/.env.example`

Neue Variable `MQTT_GATEWAY_UG65_PASSWORD`. Plus Hinweis: `MQTT_GATEWAY_BRIDGE_USER/PASSWORD` darf den `chirpstack`-User nutzen (Basic-Station-Mode aktuell nicht aktiv).

### `docs/RUNBOOK.md` §10a (neu)

Setup-Anleitung pro Server (Erst-Setup + UG65-Reconfigure + Verifikation + Rollback).

## Server-Setup-Workflow

Wird einmalig pro Server (test, main) durchgefuehrt:

1. Drei Passwoerter mit `openssl rand -hex 32` erzeugen, in `.env` eintragen.
2. `passwd`-Datei via Mosquitto-Container generieren (siehe RUNBOOK §10a.1).
3. `chmod 0640 infra/mosquitto/config/passwd`.
4. `docker compose up -d --force-recreate mosquitto chirpstack chirpstack-gateway-bridge api`.
5. UG65 Web-UI: User-Credentials einschalten + Username `gateway-ug65` + Passwort + Reboot.
6. Verifikation: Mosquitto verweigert anonymous-Verbindungen, ChirpStack-UI zeigt UG65 als online.

## Edge Cases

- `MQTT_GATEWAY_BRIDGE_USER/PASSWORD` zeigt aktuell auf `chirpstack`-Credentials (Basic-Station-Bridge ist im aktuellen Setup nicht aktiv). Wenn spaeter ein zweiter Gateway-Typ via Basic-Station hinzukommt: separater User + ACL-Eintrag.
- `passwd`-Datei ist gitignored. Pro Server eigenstaendig generieren — keine Sync-Logik.
- Healthcheck nutzt `MQTT_HEIZUNG_PASSWORD` per Compose-env-Var. Compose-Variable-Interpolation greift, kein doppeltes Dollar noetig.

## Risiken

| Risiko | Eintritt | Impact | Mitigation |
|---|---|---|---|
| `passwd` falsch generiert -> Mosquitto crasht | mittel | hoch | RUNBOOK-Anleitung Schritt fuer Schritt + Verifikations-Sequenz |
| UG65-Credentials nicht synchron mit `passwd` | hoch (manueller Schritt) | hoch (UG65 disconnected) | RUNBOOK §10a.2 + Verifikations-Subscribe |
| ChirpStack-Container kann nicht zu Mosquitto connecten | mittel (envsubst-Issue) | hoch | chirpstack-init rendert TOML mit env-Vars, gleicher Mechanismus wie bisher |
| Backend MQTT-Subscriber re-connect-loop | mittel | mittel | aiomqtt hat eigene Reconnect-Logik, einmal user/pass-Korrektur dann stabil |

## Verifikation nach Deploy

```bash
# Mosquitto verweigert anonymous
sudo docker exec deploy-mosquitto-1 mosquitto_sub -h 127.0.0.1 -p 1883 -t '$SYS/#' -C 1 -W 3
# Erwartung: "Connection Refused: not authorised"

# Mit korrektem User: OK
set -a; . /opt/heizung-sonnblick/infra/deploy/.env; set +a
sudo docker exec deploy-mosquitto-1 \
  mosquitto_sub -h 127.0.0.1 -p 1883 -u heizung-api -P "$MQTT_HEIZUNG_PASSWORD" -t '$SYS/#' -C 1 -W 3
# Erwartung: ein $SYS-Topic

# UG65 publisht (nach Reconfigure)
sudo docker exec deploy-mosquitto-1 \
  mosquitto_sub -h 127.0.0.1 -p 1883 -u heizung-api -P "$MQTT_HEIZUNG_PASSWORD" \
  -t 'eu868/gateway/+/event/stats' -C 1 -W 60
# Erwartung: Stats-Frame innerhalb 30-60 s
```

## Nicht enthalten (Backlog)

- **TLS auf MQTT** (`mqtts://`, Port 8883) — fuer Internet-exposed Brokers Best Practice. Aktuell akzeptabel weil Broker nur auf Public-IP des Hetzner-Servers laeuft + Auth aktiv. Sprint 9 oder spaeter.
- **Zertifikatsbasierte Auth** statt passwd. Overhead fuer aktuell 1 Gateway nicht gerechtfertigt.
- **Per-Tenant-Topic-Scoping** — bei Multi-Hotel-Erweiterung relevant. Aktuell 1 Tenant.
